สวัสดีครับ/ค่ะ

ขอเริ่มด้วยคำถามครับ/ค่ะ: ถ้าผู้จัดการดีลเลอร์นั่งลงในวันที่ 1 ของเดือนเพื่อตัดสินใจว่าจะสั่งรถกี่คัน เขาหรือเธอจะใช้ตัวเลขอะไร?

คำตอบที่ได้ยินบ่อยที่สุดคือสามัญสำนึก ตัวเลขปีที่แล้ว หรือ spreadsheet ที่วาดเส้นตรงง่ายๆ ปัญหาของทั้งสามวิธีคือมันมองข้ามปฏิสัมพันธ์ที่ขับเคลื่อนอุปสงค์จริงๆ ไม่ว่าจะเป็นโปรโมชั่นที่กำลังดำเนินอยู่ในเดือนนั้น ภูมิภาคที่สินค้าจะไป หรือส่วนลดของคู่แข่งที่เปลี่ยนไป

โครงการนี้สร้างและสาธิตระบบที่แทนที่สามัญสำนึกด้วยการพยากรณ์จาก Machine Learning ที่มีความแม่นยำภายใน 7 เปอร์เซ็นต์ ขอนำเสนอให้ฟังครับ/ค่ะ

## หัวข้อที่ 1 — ปัญหาและบริบทธุรกิจ (~2 นาที)

**[คำแนะนำ: ยังไม่ต้องแตะ Dashboard คือช่วงสร้างบริบท]**

---

ปัญหาทางธุรกิจคือการพยากรณ์ความต้องการสำหรับกลุ่มยานยนต์หลายแบรนด์ที่ดำเนินงานใน 5 ภูมิภาคทั่วประเทศไทย

ทำไมถึงสำคัญทางการเงิน?

**สต็อกเกิน:** สินค้าคงคลังที่เกินผูกเงินทุนหมุนเวียน รถที่จอดอยู่ในโชว์รูม 90 วัน มีต้นทุนทางการเงินประมาณ 1.5% ของมูลค่าต่อเดือน สำหรับรถราคาเฉลี่ย 750,000 บาท นั่นคือ 11,000 บาทต่อคันต่อเดือน

**สต็อกขาด:** หมายถึงยอดขายที่สูญไปและกำไรที่หายไป ไม่มีโอกาสครั้งที่สองเมื่อลูกค้าเดินออกไปซื้อจากคู่แข่ง

การพยากรณ์ที่แม่นยำขึ้นแม้เพียง 3 เปอร์เซ็นต์จุดใน 2,100 การรวมกันของแบรนด์-ภูมิภาค-เดือน แปลงตรงสู่ความเสี่ยงสินค้าคงคลังที่น้อยลงและยอดขายที่พลาดน้อยลง

ความท้าทายรองคือวิธีการแบบง่ายล้มเหลวในขนาดนี้ ARIMA ซึ่งเป็นโมเดลอนุกรมเวลาแบบคลาสสิก มองเห็นได้เฉพาะประวัติยอดขายของชุดข้อมูลเดียว ไม่สามารถรู้ได้ว่ากำลังมีโปรโมชั่น หรือว่าภาคกรุงเทพฯ มีประสิทธิภาพสูงกว่าภาคตะวันออกเฉียงเหนือถึง 75% เราจำเป็นต้องใช้โมเดลที่รับข้อมูลหลายตัวแปรพร้อมกัน

## หัวข้อที่ 2 — ข้อมูลและแนวทางสร้างโมเดล (~2 นาที)

**[คำแนะนำ: คลิกไปที่ Tab 1 — Overview ให้ผู้ฟังเห็นข้อมูลขณะพูด]**

---

[เปิด Tab 1 — Overview & EDA]

ชุดข้อมูลครอบคลุมตั้งแต่มกราคม 2561 ถึงธันวาคม 2567 รวม 84 เดือน สำหรับ 5 แบรนด์ ได้แก่ Toyota, Honda, Isuzu, Ford และ Mazda และ 5 ภูมิภาค ทำให้ได้ 2,100 การสังเกตในระดับแบรนด์-ภูมิภาค-เดือน

ข้อมูลใช้แบบจำลองอุปสงค์แบบ multiplicative ยอดขายในแต่ละเดือนเท่ากับค่าพื้นฐานของแบรนด์-ภูมิภาค คูณด้วยแนวโน้มระยะยาว ปัจจัยฤดูกาล ตัวคูณโปรโมชั่น และความผันผวน

[ชี้ที่กราฟฤดูกาล]

คุณจะเห็นรูปแบบฤดูกาลได้ชัดเจน ธันวาคมและมีนาคมเป็นเดือนที่ยอดขายสูงสุด เนื่องจากการต่ออายุยานพาหนะของบริษัทช่วงสิ้นปี และการเปิดตัวรุ่นใหม่

[ชี้ที่กราฟผลของโปรโมชั่น]

นี่คือผลของโปรโมชั่น เดือนที่มีโปรโมชั่นเฉลี่ยขายได้มากกว่าเดือนที่ไม่มีโปรโมชั่นประมาณ 15% โมเดลใดก็ตามที่ไม่สามารถคำนึงถึงสิ่งนี้ จะทำนายต่ำกว่าความเป็นจริงอย่างมีระบบในช่วงโปรโมชั่น

เราสร้างและเปรียบเทียบ 3 โมเดล: ARIMA เป็นเส้นฐานแบบคลาสสิก จากนั้น XGBoost และ LightGBM ซึ่งเป็นโมเดล gradient-boosted tree เป็นแนวทางหลัก เนื่องจากโมเดล tree สามารถใช้ข้อมูล 17 ตัวแปรพร้อมกัน และจับความสัมพันธ์แบบไม่เชิงเส้นได้ เช่น การที่โปรโมชั่นในธันวาคมมีค่ามากกว่าโปรโมชั่นในเดือนสิงหาคมมาก

## หัวข้อที่ 3 — ผลลัพธ์ของโมเดล (~2 นาที)

**[คำแนะนำ: คลิกไปที่ Tab 3 — Model Comparison อยู่ที่นี่เพื่อดูตัวเลข]**

---

[เปิด Tab 3 — Model Comparison]

ขอให้ตัวเลขหลักกันก่อน

ARIMA ทำได้ MAPE — ค่าความคลาดเคลื่อนเปอร์เซ็นต์สัมบูรณ์เฉลี่ย — ที่ 2.2% สำหรับชุดข้อมูลตลาดรวมทั้งหมด ดูน่าประทับใจ

XGBoost ทำได้ MAPE 7.2% และ LightGBM ได้ 7.2% เช่นกัน

ตัวเลขเหล่านี้ดูเหมือนจะบอกว่า ARIMA ชนะ แต่มีข้อแม้สำคัญ: ARIMA ทำงานกับชุดข้อมูลรวมที่เรียบขึ้น ซึ่งก็คือแบรนด์ทั้งหมด ภูมิภาคทั้งหมด รวมกัน มันไม่สามารถบอกได้ว่าควรสั่ง Toyota กี่คันสำหรับกรุงเทพฯ หรือ Isuzu กี่คันสำหรับภาคตะวันออกเฉียงเหนือ ความละเอียดนั้นต้องใช้ XGBoost

เมื่อเปรียบเทียบในระดับเดียวกัน ซึ่งก็คือระดับแบรนด์-ภูมิภาคแบบละเอียด ค่าความคลาดเคลื่อน 7% ของ XGBoost คือตัวชี้วัดที่เกี่ยวข้อง ซึ่งคลาดเคลื่อนประมาณ 28 คันต่อแบรนด์-ภูมิภาค-เดือน จากฐานประมาณ 350 คัน นั่นแม่นยำพอสำหรับการตัดสินใจสั่งซื้อจริง

[ชี้ที่กราฟ Feature Importance]

กราฟ Feature Importance บอกว่าโมเดลเรียนรู้อะไร ยอดขายย้อนหลัง 12 เดือน (lag_12) หรือยอดขายในเดือนเดียวกันของปีที่แล้ว เป็นตัวทำนายที่แข็งแกร่งที่สุด ฤดูกาลรายปีคือสัญญาณหลัก ค่าเฉลี่ยเคลื่อนที่ 6 เดือนและ lag 1 เดือนอยู่ถัดมา การจับแรงผลักดันระยะสั้น โปรโมชั่นอยู่กลางกราฟ ยืนยันว่าโมเดลเรียนรู้การเพิ่มขึ้นที่เราเห็นใน EDA

## หัวข้อที่ 4 — สาธิต Dashboard (~5 นาที)

**[คำแนะนำ: นี่คือการสาธิตสด ไล่ผ่าน Tab 1 → 2 → 3 → (Tab 4 เป็นหัวข้อถัดไป) ให้การโต้ตอบสั้นๆ — hover, filter ครั้งเดียว แล้วไปต่อ]**

---

[เปิด Tab 1 — Overview & EDA]

ขอพาชมผ่าน Dashboard ซึ่งนี่คือสิ่งที่ผู้ใช้ธุรกิจจะโต้ตอบในแต่ละวัน ไม่ต้องใช้ Python ไม่ต้องเขียนโค้ด

แถบด้านซ้ายมีตัวกรอง 3 ตัว: แบรนด์, ภูมิภาค และช่วงปี ตัวกรองเหล่านี้ใช้ได้กับทุก tab พร้อมกัน

[ในแถบด้านข้าง ยกเลิกการเลือกทุกแบรนด์ แล้วเลือกเฉพาะ Toyota]

ถ้าผมกรองเฉพาะ Toyota ทุกกราฟในหน้านี้อัปเดตทันที ทั้งเส้นแนวโน้ม รูปแบบฤดูกาล และการแบ่งตามภูมิภาค

[ชี้ที่กราฟภูมิภาค]

กรุงเทพฯ นำ ตามที่คาดไว้ แต่ภาคเหนือและภาคตะวันออกเฉียงเหนือไม่ไกลนัก เพราะ Toyota ขายกระบะเป็นสัดส่วนมากให้กับลูกค้าในภาคเกษตรกรรม

[เปิด Tab 2 — Forecast]

Tab 2 คือมุมมองการพยากรณ์ สามารถเลือก ARIMA เพื่อดู confidence interval ของตลาดรวม บริเวณแรเงาคือ confidence interval 95% ซึ่งขยายกว้างขึ้นยิ่งพยากรณ์ไกลออกไปในอนาคต

[เปลี่ยนไปใช้ XGBoost ในปุ่มวิทยุ]

หรือเปลี่ยนเป็น XGBoost ซึ่งแสดงประสิทธิภาพจริงปี 2567 เทียบกับการพยากรณ์ของโมเดล เส้นสีเขียวคือสิ่งที่เกิดขึ้นจริง เส้นประสีแดงคือสิ่งที่โมเดลพยากรณ์ ติดตามกันได้ใกล้ชิด

[เลือก "Toyota" จาก dropdown แบรนด์]

สามารถเจาะลงไปที่แบรนด์แต่ละตัวได้ นี่คือ Toyota สำหรับปี 2567 โมเดลจับการพุ่งขึ้นในธันวาคมและการลดลงช่วงกลางปีได้อย่างถูกต้อง

[เปลี่ยนแถบด้านข้างกลับไปที่แบรนด์ทั้งหมด]

Tab 3 ซึ่งเราดูไปแล้ว มีการเปรียบเทียบตัวชี้วัดแบบเคียงข้างกัน และกราฟวินิจฉัย ฮิสโตแกรม residual แสดงว่า error กระจายสมมาตรและอยู่ที่ศูนย์ ซึ่งบอกว่าไม่มี bias อย่างเป็นระบบ

## หัวข้อที่ 5 — การดูแลรักษาโมเดล (~1.5 นาที)

**[คำแนะนำ: เปิด Tab 4 — Model Maintenance]**

---

[เปิด Tab 4 — Model Maintenance]

การสร้างโมเดลเป็นแค่ครึ่งหนึ่งของงาน อีกครึ่งหนึ่งคือการรักษาความน่าเชื่อถือเมื่อเวลาผ่านไป Tab นี้ตอบคำถามที่ทุก deployment ต้องเผชิญในท้ายที่สุด: รู้ได้อย่างไรว่าโมเดลหยุดทำงานดีแล้ว?

[ชี้ที่กราฟแนวโน้ม MAPE]

เส้นสีน้ำเงินทึบแสดง MAPE รายไตรมาสของโมเดลตลอดปี 2567 ยังดี อยู่ต่ำกว่าขีดจำกัด watch 10% มาก

เส้นประสีส้มเป็นการจำลองว่าจะเกิดอะไรขึ้นถ้าไม่มีการ retrain เลย เมื่อสภาวะตลาดเปลี่ยนแปลง MAPE จะค่อยๆ ขยับขึ้นผ่าน watch zone และข้าม retrain threshold 15% ในที่สุด

[ชี้ที่กราฟฮิสโตแกรม drift]

ด้านล่างเราเปรียบเทียบการกระจายตัวของ feature ระหว่างช่วงการฝึก 2561–2565 กับข้อมูลล่าสุดปี 2567 ถ้าการกระจายทับกันดี โมเดลกำลัง score ข้อมูลที่ดูคล้ายกับสิ่งที่ถูก train มา ส่วนต่างขนาดใหญ่คือสัญญาณเตือนล่วงหน้า

[ชี้ที่ workflow cards]

และที่ด้านล่าง workflow การ retrain 6 ขั้นตอน: ตรวจจับ, เก็บข้อมูลใหม่, ฝึกใหม่, ตรวจสอบ, deploy ใหม่, reset baseline ในระบบ production ลูปนี้สามารถทำอัตโนมัติได้ส่วนใหญ่ด้วย scheduled pipeline บน AWS SageMaker หรือ Apache Airflow

## หัวข้อที่ 6 — คุณค่าทางธุรกิจและสรุป (~1 นาที)

**[คำแนะนำ: ปิด Dashboard หรือถอยออกจากหน้าจอ สบตากับผู้ฟังโดยตรงสำหรับการปิด]**

---

สรุปนะครับ/ค่ะ

เราแทนที่โมเดลสถิติแบบตัวแปรเดียวด้วยระบบ gradient boosting แบบหลายตัวแปรที่ละเอียดพอสำหรับตัดสินใจสั่งซื้อจริงในระดับแบรนด์-ภูมิภาค มันรับรู้โปรโมชั่น ฤดูกาล รูปแบบอุปสงค์ระดับภูมิภาค และโมเมนตัมปีต่อปีพร้อมกัน

Pipeline ไม่ใช่การวิเคราะห์ครั้งเดียว แต่เป็นระบบที่ดูแลรักษาได้ Tab monitoring บอกทีม operations ว่าควรดำเนินการเมื่อใดก่อนที่ลูกค้าจะสังเกตเห็นว่าโมเดลล้าสมัย

ผลกระทบในทางปฏิบัติตรงไปตรงมา: กลุ่มดีลเลอร์ที่สั่ง 2,100 การรวมกันต่อเดือนด้วยความคลาดเคลื่อนในการพยากรณ์ 7% แทนที่จะเป็น 15% จากสัญชาตญาณ มีความเสี่ยงสินค้าคงคลังน้อยกว่า stockout น้อยกว่า และ cash conversion ดีกว่า

ขอบคุณมากครับ/ค่ะ ยินดีรับคำถาม

### คำถามที่ 1: ทำไมถึงใช้ข้อมูลจำลองแทนข้อมูลดีลเลอร์จริง?

**คำตอบ:** ข้อมูลธุรกรรมดีลเลอร์จริงเป็นข้อมูลที่เป็นความลับและอยู่ภายใต้ NDA การจำลองให้สิ่งที่ดีกว่าสำหรับการสาธิตระเบียบวิธี: ground truth ที่ทราบค่า เราเข้ารหัส promotion uplift 15%, แนวโน้ม 2% ต่อปี และยอดขายสูงสุดในธันวาคม จากนั้นสามารถตรวจสอบว่าโมเดล recover parameter เหล่านั้นได้จริงๆ ซึ่งมันทำได้ Pipeline เดียวกันทำงานได้ไม่เปลี่ยนแปลงกับข้อมูลจริงเมื่อมีให้ใช้

### คำถามที่ 2: ARIMA ได้ MAPE ต่ำกว่า XGBoost นั่นหมายความว่า ARIMA ดีกว่าหรือไม่?

**คำตอบ:** ไม่ใช่ และนี่คือความแตกต่างด้านระเบียบวิธีที่สำคัญที่สุด ARIMA ทำงานบนชุดข้อมูลรวมทั้งตลาด: ตัวเลขเรียบๆ ตัวเดียวต่อเดือน XGBoost ทำงานบนชุดข้อมูล 2,100 ชุดแต่ละแบรนด์-ภูมิภาค การรวมข้อมูลลดความแปรปรวน ดังนั้น MAPE 2% ของ ARIMA บนเส้นเรียบไม่สามารถเปรียบเทียบได้กับ MAPE 7% ของ XGBoost บนชุดข้อมูลละเอียด สำหรับการตัดสินใจสั่งซื้อจริง คุณต้องการการพยากรณ์ที่ละเอียด ARIMA ไม่สามารถบอกได้ว่า Toyota กรุงเทพฯ เทียบกับ Ford ภาคตะวันออกเฉียงเหนือ

### คำถามที่ 3: จะจัดการกับแบรนด์ใหม่หรือภูมิภาคใหม่ที่ไม่อยู่ในข้อมูลการฝึกอย่างไร?

**คำตอบ:** โมเดล tree ไม่สามารถ extrapolate ไปยัง category ที่ไม่เคยเห็น นี่เป็นข้อจำกัดที่ทราบกัน วิธีแก้ในทางปฏิบัติคือ cold-start rule: สำหรับ 6–12 เดือนแรกของแบรนด์/ภูมิภาคใหม่ ให้ใช้ baseline ตามกฎง่ายๆ (เช่น ค่าเฉลี่ยภูมิภาคสำหรับระดับแบรนด์) ขณะที่เก็บข้อมูลให้เพียงพอสำหรับโมเดลที่จะเรียนรู้ชุดข้อมูลนั้น หลังจาก 12 การสังเกต มีข้อมูลเพียงพอที่จะรวมไว้ในรอบ retraining ถัดไป

### คำถามที่ 4: ต้นทุนของการพยากรณ์ที่ผิดคืออะไร? วัดผลกระทบทางธุรกิจอย่างไร?

**คำตอบ:** มีต้นทุน 2 ด้านที่ไม่สมมาตรกัน การพยากรณ์สูงเกินไปทำให้ overstock: ต้นทุนการเงินประมาณ 1.5% ต่อเดือนของมูลค่ายานพาหนะ ดังนั้นรถ 750,000 บาทที่ขายไม่ออก 1 เดือน มีต้นทุนประมาณ 11,000 บาท การพยากรณ์ต่ำเกินไปทำให้ stockout: กำไรที่สูญเสียจากรถ Sedan ระดับกลางอยู่ที่ประมาณ 30,000–50,000 บาท เนื่องจากต้นทุน stockout สูงกว่า จึงควร bias การพยากรณ์ขึ้นเล็กน้อยสำหรับรุ่นที่มีกำไรสูงในช่วง peak season นี่คือ business calibration layer บนโมเดล

### คำถามที่ 5: ทำไมถึงเลือก XGBoost แทน neural network หรือ transformer?

**คำตอบ:** สามเหตุผล ประการแรก ข้อมูลตาราง (tabular) ที่มี feature engineering (lags, rolling means) คือที่ที่ gradient boosting ตีเสมอหรือชนะ deep learning ได้อย่างสม่ำเสมอ มีหลักฐานอ้างอิงมากมายใน Kaggle literature ประการที่สอง interpretability: feature importance จาก XGBoost นำไปปฏิบัติได้ทันทีสำหรับผู้ฟังทางธุรกิจ ประการที่สาม เวลาการฝึก: XGBoost ฝึกในเวลาไม่ถึง 10 วินาทีที่นี่ เทียบกับนาทีถึงชั่วโมงสำหรับ transformer โดยไม่ต้องใช้ GPU สำหรับขนาดข้อมูลและโครงสร้างนี้ XGBoost คือเครื่องมือที่เหมาะสม

### คำถามที่ 6: พร้อมใช้งานจริง (production-ready) มากแค่ไหน?

**คำตอบ:** เวอร์ชันปัจจุบันเป็น proof of concept ระดับ portfolio inference script (inference.py) ใช้ SageMaker 4-function protocol และสามารถ deploy บน real endpoint ได้ด้วย 2 คำสั่ง monitoring module ทำงานอิสระและส่งออก JSON summary ที่สามารถ trigger Airflow DAG ได้ ในการใช้งานจริงต้องการ: real data pipeline แทน CSV, IAM role setup และขั้นตอน CI/CD เพื่อ promote โมเดลหลัง validation ผ่าน
